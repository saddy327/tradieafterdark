
# TradieAfterDark — Full Replit Build Prompt (v2)

Build a full-stack web application called **TradieAfterDark** — a subscription-based marketplace connecting busy professionals who need after-hours trade services with tradies looking to take on work outside standard business hours.

The domain is `tradieafterdark.com`. The brand tone is bold, dark, and no-nonsense — built for people who get things done when others are asleep. Trust, verification, and licensing rigour are the product's competitive moat. Every design decision should reinforce that.

---

## POSITIONING

**Supply side:** Qualified, fully licensed trade contractors who already operate (or are eligible to operate) their own ABN-registered side work and want a clean channel for after-hours leads. Not apprentices. Not unlicensed workers.

**Demand side:** Time-poor professionals, landlords, and small business owners who need trade work done outside of 9–5 and are willing to pay a small premium for after-hours availability and verified credentials.

**The promise to customers:** *"Every tradie on this platform is licence-verified, ID-verified. We've checked, so you don't have to."*

---

## TECH STACK

- **Frontend:** React + TypeScript + Vite + Tailwind CSS
- **Backend:** Node.js + Express + TypeScript
- **Database:** PostgreSQL (via Prisma ORM) — provision with PostGIS extension enabled from day one for future geo search
- **Auth:** JWT access tokens (15 min) + rotating refresh tokens (30 days), both in httpOnly cookies. CSRF token via double-submit pattern.
- **Payments:** Stripe (subscriptions, billing portal, webhook idempotency table)
- **File Storage:** Cloudinary using **signed upload presets** — never expose API secret to the client
- **Email:** Postmark (transactional) — Nodemailer/SMTP only as a local-dev fallback. SPF, DKIM, DMARC configured before launch.
- **Scheduling:** node-cron for licence expiry and job-completion review prompts
- **Deployment-ready:** `.env` config, seed script, full README

## USER ROLES

### 1. Tradie (Service Provider)
- Pays a monthly or annual subscription to be listed
- Must complete licence verification, ID verification, and insurance declaration before going live
- Manages leads in-platform (messaging stays on-platform until both parties exchange contact details)
- Keeps 100% of what they charge — TradieAfterDark takes no commission
- Can mark themselves as available/unavailable on a per-day basis

### 2. Customer (Service Seeker)
- Free to sign up and browse
- Searches tradies by trade, suburb/postcode (with radius), and availability
- Books jobs through the platform — every job has a lifecycle the platform tracks
- Leaves reviews **only on completed jobs they actually booked** (one review per completed job)

### 3. Admin (Internal)
- Verification queue management (licence + ID + insurance)
- Subscription and account management
- Complaints and dispute moderation
- Revenue and operational dashboards
- Admin accounts are seeded only — no public admin signup

## KEY PAGES & FEATURES

### Public / Marketing

**Homepage** — Hero with tagline *"When the sun goes down, we get to work."* Three-step "How it works", featured trades, trust strip showing the verification process (ID + Licence + Insurance), customer testimonials section (placeholder), tradie acquisition CTA.

**Browse Tradies** — Search bar (trade + postcode), filters (evenings / weekends / both,) Only profiles where `isLive === true` are listed. Paginated, mobile-first.

**Pricing page** — see Stripe section.

**Trust & Verification page** — explains exactly how verification works. This is marketing for the moat.

### Auth

- **/signup** — Two paths: "I need a tradie" (customer) and "I am a tradie" (subscribe + list).
- **/login** — Email + password with forgot-password flow (rate-limited, single-use signed tokens, 1-hour expiry).
- Refresh token rotation: every refresh issues a new refresh token and invalidates the previous one. Reuse of an invalidated refresh token revokes the entire family.

### Tradie Onboarding (multi-step)

1. **Identity** — full legal name, DOB, mobile, residential suburb/state, ABN (required — entered, validated against ABN format, ABN status checked via the public ABN Lookup API where possible). Upload of government-issued photo ID (driver's licence or passport).
2. **Trade & service area** — trade category (dropdown). Service postcodes (multi-select) with optional radius from a central postcode. Hourly rate, bio, years of experience.
3. **Availability** — evenings / weekends / both, plus a per-day-of-week toggle.
4. **Licence** — for licensed trades: licence number, licence class, issuing state (drives the issuing authority via server-side `STATE_AUTHORITY_MAP`), expiry date, document upload (PDF/image, max 5MB, signed Cloudinary upload). Multiple licences can be added if the tradie offers multiple licensed trades — each licence is verified independently. For unlicensed trades, a Statutory Declaration checkbox replaces the licence form.
5. **Insurance** — public liability insurance details: insurer name, policy number, cover amount (minimum $1M recommended), expiry date, certificate of currency upload. **A tradie can list without insurance, but their public profile will display "Not insured" prominently and they cannot be marked "Verified".**
6. **Subscription checkout** — Stripe Checkout.

A profile only goes live when **all** of the following are true:
- Payment confirmed (Stripe webhook)
- ID verified by admin
- Every claimed licensed trade has `verificationStatus = VERIFIED` (or `SELF_DECLARED` for unlicensed trades)
- Insurance is either VERIFIED or the tradie has explicitly opted in to listing without insurance

After step 6 the tradie sees: *"Your application is under review. We verify ID, licence, and insurance within 1 business day. We'll email you when you're live."*

## STRIPE INTEGRATION

- Two products on setup: `monthly_plan` ($49/month), `annual_plan` ($412/year)
- Onboarding step 6 redirects to Stripe Checkout
- Webhook handler at `/api/webhooks/stripe`:
  - **Idempotency:** every event ID is recorded in a `StripeWebhookEvent` table on first processing. Duplicates are no-ops.
  - `checkout.session.completed` → mark payment confirmed; trigger live-eligibility check
  - `customer.subscription.deleted` / `customer.subscription.paused` → set `isLive = false`
  - `invoice.payment_failed` → email tradie, flag account, set `isLive = false` after grace period (7 days)
  - `customer.subscription.updated` → sync `subscriptionStatus`
- **Refund policy on rejected verification:** if a tradie pays but is rejected at verification, they get one free resubmission window (14 days). If they don't resubmit or fail again, the subscription is cancelled and prorated refund issued via Stripe. This must be documented in the ToS and surfaced in the rejection email.
- Tradie dashboard links to Stripe Customer Portal for self-service billing

## TRADIE LICENCE VERIFICATION SYSTEM

### Australian Licensing Authorities by State

| State | Authority | Public Search URL |
|---|---|---|
| QLD | QBCC | qbcc.qld.gov.au/licensing-certification/search-qbcc-licence-register |
| NSW | NSW Fair Trading | onlineregistry.fairtrading.nsw.gov.au |
| VIC | VBA | vba.vic.gov.au/tools/licence-register |
| SA | CBS | cbs.sa.gov.au/resources/licence-register |
| WA | Building Commission WA | buildingcommission.wa.gov.au/licence-register |
| TAS | CBOS | cbos.tas.gov.au |
| ACT | Access Canberra | accesscanberra.act.gov.au |
| NT | NT Building Advisory Services | nt.gov.au/industry |

Federal: ARCtick (refrigerant handling) — `arctick.org` — required in addition to state HVAC licence.

Store the authority name and lookup URL pattern per state in `STATE_AUTHORITY_MAP` server-side.

### Trade-to-Licence Map (Per State)

The mapping is keyed by **(state, trade)** because licence classes vary materially across states. Treat the structure below as the canonical shape — populate authoritative values during build by checking each state authority directly.

```typescript
// server/config/licenceMap.ts
type LicenceRequirement = {
  authority: string;
  licenceClassOptions: string[]; // valid licence class strings for this state+trade
  unlicensedThresholdAud?: number; // dollar threshold below which no licence is required
  requiresInsurance: boolean;
};

const LICENCE_MAP: Record<State, Record<Trade, LicenceRequirement>> = {
  QLD: {
    Electrician: { authority: "QBCC + Electrical Safety Office", licenceClassOptions: ["Electrical Contractor", "Electrical Worker (with QBCC contractor licence)"], requiresInsurance: true },
    Plumber:     { authority: "QBCC", licenceClassOptions: ["Plumbing & Drainage Contractor"], requiresInsurance: true },
    Builder:     { authority: "QBCC", licenceClassOptions: ["Builder - Low Rise", "Builder - Medium Rise", "Builder - Open"], requiresInsurance: true },
    // ... etc
    Handyman:    { authority: "None below threshold", licenceClassOptions: [], unlicensedThresholdAud: 3300, requiresInsurance: false },
  },
  NSW: {
    Handyman:    { authority: "None below threshold", licenceClassOptions: [], unlicensedThresholdAud: 5000, requiresInsurance: false },
    // ... etc
  },
  // ... all states
};
```

**Server-side enforcement rule:** A tradie can only list under trade categories that match a `VERIFIED` licence (or are explicitly unlicensed in their state). Adding a trade without a matching verified licence is rejected at the API layer with a 403, regardless of what the frontend allows. This is also covered by the `ProfileChangeLog` model below.

### Identity Verification

Separate from licence verification. Admin reviews the uploaded ID document and confirms:
1. The document is genuine on its face (not obviously photoshopped)
2. The name on the ID matches the name on the licence
3. The name on the ID matches the account holder

Without ID verification, "verified" only proves a licence exists — not that this user holds it.

### Insurance Verification

Admin reviews uploaded certificate of currency, confirms insurer, policy number, cover amount, and expiry. A tradie's `insuranceVerificationStatus` is independent of licence status.

### Cron Jobs (daily, `Australia/Brisbane` TZ)

- Licences expiring in 30 days → warn tradie via email
- Licences expired → set status `EXPIRED`, set `isLive = false`, email tradie
- Insurance certificates expiring in 30 days → warn tradie
- Insurance expired → strip "Insured ✓" badge, email tradie (does not auto-take-down profile, but flags it)
- Jobs marked `IN_PROGRESS` for >14 days → prompt customer to mark complete or open dispute
- Jobs marked `COMPLETED` >24h ago without review → send review prompt

### Statutory Declaration (Unlicensed Trades)

For unlicensed trade types in the relevant state:
- Replace licence upload with a checkbox + timestamp + IP capture: *"I declare that the services I am offering do not require a contractor's licence under applicable Australian law in [STATE] and that I am legally permitted to carry out this work."*
- Status set to `SELF_DECLARED`, auto-approved, no admin review
- Public profile shows grey "Self-Declared" badge — visually distinct from orange "Verified"
- Admin can flag and remove on complaint

## IN-APP MESSAGING & JOB LIFECYCLE

The platform retains conversation context until both parties have a structured reason to take it off-platform.

- Customer creates a job request → status `ENQUIRY`, message thread starts
- Tradie can accept → status `ACCEPTED`, contact details revealed to both parties in a structured "contact card" within the thread
- Tradie marks `IN_PROGRESS` when work begins
- Either party marks `COMPLETED`; the other confirms within 7 days or it auto-confirms
- Review prompt fires 24h after completion
- Either party can open a `Dispute` at any post-`ENQUIRY` status

**Contact-info leak detection in messages (pre-acceptance):** before `ACCEPTED`, run a regex pre-flight on outbound messages for phone numbers / emails / "WhatsApp me" patterns. Don't block — flag for moderation and show a soft warning to the sender: *"You can share contact details once you've accepted this job."* The point isn't to police adults; it's to keep enough data on-platform that ranking and reviews work.

## SECURITY, COMPLIANCE, OPERATIONAL

- **Rate limiting** on: signup, login, password reset, enquiry/job creation, message send, Stripe webhook, admin endpoints. Use a sliding window via Redis (or in-memory for MVP, swap later).
- **Input validation** server-side using Zod schemas shared with the client where possible. Sanitise all rich-text fields. Strip EXIF from uploaded images.
- **Raw licence numbers, raw insurance policy numbers, raw ABNs** are never returned by any public endpoint — only to the owner and admins.
- **Australian Privacy Act compliance:** capture consent timestamps, provide a data export endpoint and a data deletion request flow (soft-delete with 30-day retention before hard delete).
- **Cloudinary uploads** use signed presets generated server-side per request. Never embed the API secret in the client.
- **CSRF protection** via double-submit cookie pattern on all state-changing requests.
- **Audit log** every admin action (verification decision, suspension, dispute resolution).
- **Soft-delete** on `User`, `TradieProfile`, `PortfolioImage`, `Review` — never hard-delete records that may be needed for record-keeping.
- **Slug strategy:** generate from displayName, lowercase, hyphenate. On collision, append a 4-char random hash (not a numeric suffix) so freed slugs aren't reassigned.
- **Webhook idempotency** via the `StripeWebhookEvent` table.
